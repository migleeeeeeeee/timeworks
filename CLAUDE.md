# Design System — Project Context

A solo-built design system delivered as three connected artifacts: a Figma library, a Storybook, and a coded React + TypeScript component library.
The existing app this serves is being migrated onto the system component-by-component.

**Status (2026-04-30):** v1 is shipped. 42 components live in `src/components/` (well beyond the original 12-component plan). Token pipeline is wired, three themes are running (light/dark/black). The current focus is hardening — visual QA across themes, removing the last hardcoded values, and adding tests.

---

## Stack

- **React 18** + **TypeScript** (strict mode, `noUncheckedIndexedAccess: true`)
- **Vite** for dev server and build
- **Tailwind v4** with CSS variables (tokens are CSS vars consumed via `@theme`)
- **Radix UI** primitives as the base for behavior and accessibility
- **class-variance-authority (cva)** for variant styling
- **clsx + tailwind-merge** via a `cn()` helper for conditional classes
- **lucide-react** for icons (re-exported under our own names from `src/icons/`)
- **Storybook 8** (React-Vite framework) for documentation, with `addon-a11y`, `addon-themes`, `addon-interactions`
- **Tokens Studio** (Figma plugin) is the source of truth for tokens. Designers edit Figma variables, push the JSON export to `src/tokens/source/`, and `scripts/build-tokens.mjs` compiles it to `src/tokens/dist/css/variables.css`.

**Tokens flow Figma → code.** Designers change Figma variables in Tokens Studio, the export lands in `src/tokens/source/`, and `npm run tokens:build` regenerates the CSS variables consumed by every component. Component _authoring_ also flows Figma → code (see "Authoring workflow").

## Tokens

The pipeline is real and runs every time `src/tokens/source/` changes:

1. **Designer edits in Figma** — Tokens Studio plugin manages the Figma variables.
2. **Export to repo** — Tokens Studio writes JSON to `src/tokens/source/`.
3. **Build** — `npm run tokens:build` runs `scripts/build-tokens.mjs`:
   - Reads `Color Tokens/{Light,Dark,Black}.json`, synthesizes `--color-{key}` for every color.
   - Reads `Global.json` for shadows (→ `--shadow-xs/sm/md/lg`) and typography composites (→ `--text-h1/h2/h3/t1/t2/t3` + `--font-heading/body/mono`), resolving alias references like `{fontSize.5}` against the primitive groups.
   - Reads `Spacing Tokens/Mode 1.json`, emits `--space-{px}` (Figma-authoritative names) **and** `--spacing-{step}` (Tailwind v4 scale overrides) so utilities like `p-4`, `gap-6`, `m-8` resolve through Figma values.
   - Emits one `@theme { ... }` block (light defaults + typography + fonts + spacing + shadows) plus `[data-theme="dark"]` and `[data-theme="black"]` override blocks containing only the deltas vs light.
4. **Components consume CSS vars** through Tailwind utilities (`bg-[var(--color-primary-color)]`, `text-t2`, `gap-4`, `shadow-md`).

**Theming.** The runtime swaps `data-theme` on `<html>` (showcase: `src/showcase/App.tsx`; Storybook: toolbar via `withThemeByDataAttribute`). All three themes are exercised and components must look correct in all of them.

**Naming.** Var names mirror Tokens Studio leaf keys verbatim — e.g. `--color-primary-color`, `--color-primary-background-hover-color`, `--color-text-color-on-primary`. Don't invent shorter aliases; Figma is the source of truth for naming.

**Editing tokens.** Edit in Figma via Tokens Studio, push, run `npm run tokens:build`. Hand-editing `src/tokens/source/*.json` is a fallback only and should be reconciled back to Figma. Never edit `src/tokens/dist/`.

## Architecture rules — do not violate

1. **Components never use raw values.** No raw hex codes, no raw px spacing, no raw radii. Every visual property comes from a token, applied via a Tailwind utility that's backed by a CSS variable.
2. **Behavior comes from Radix.** If a Radix primitive exists for the pattern (Dialog, Tooltip, Select, Checkbox, Switch, etc.), build on top of it. Don't reinvent keyboard handling, focus management, or portals.
3. **Forward refs and spread rest.** Every component forwards a ref to the root element and spreads `...rest` so consumers can extend it.
4. **Variant logic uses cva.** Not inline ternaries, not template strings. Variants and sizes are typed and exported.
5. **No component is done without a Storybook story.** Story file exists, covers the canonical use, all variants, all sizes, all states (default/hover/focus/active/disabled/loading where applicable). The story is the spec.
6. **Accessibility is non-negotiable.** Every component passes the Storybook a11y addon checks. Interactive components have visible focus rings (use `focus-visible:`). Color contrast ≥ 4.5:1 for text in all three themes.
7. **No external CSS files per component.** Tailwind classes only. Global styles live in `src/index.css` and are minimal.
8. **Per-component theming uses `data-theme={prop}` on the root, not branched colors.** If a component needs to render dark while the page is light (e.g. `<DatePicker theme="dark" />`), set `data-theme` and let semantic tokens resolve via the override blocks. Don't hand-pick hex per branch.

## API conventions

- Component names: PascalCase, descriptive. `Button`, `TextField`, `Checkbox` — not `Btn`, `Input`, `Cb`.
- Size prop: `size?: "sm" | "md" | "lg"` with `md` as default.
- Variant prop: `variant?: "primary" | "secondary" | "tertiary" | "ghost" | "destructive"` (Button-style; adapt per component).
- Boolean props: `disabled`, `loading`, `required`, `readOnly`. Use the React-standard names where they exist.
- Controlled/uncontrolled pairs: `value` + `onValueChange` for controlled, `defaultValue` for uncontrolled. Match Radix conventions exactly.
- Event handlers: `onClick`, `onChange`, `onOpenChange`. Standard React/Radix naming.
- Always export both the component and its props type: `export { Button, type ButtonProps }`.
- Always `forwardRef`. Always type the ref correctly (`HTMLButtonElement`, `HTMLInputElement`, etc.).
- Always spread `...rest` to the root element so consumers can pass `aria-*`, `data-*`, `id`, etc.
- When the component wraps a Radix primitive, mirror Radix's prop API rather than inventing a new one.
- When the component has no Radix equivalent, mirror existing in-repo conventions — match the patterns of the most similar shipped component.

## Brand foundations

- **Primary hue:** indigo-violet, anchor `#6F54EB` (sourced from the Figma file, page "■ Colors", node 46814:454).
- **Typography:** **Montserrat** for headings (weights 300/400/500/600/700) and **Karla** for body and UI (weights 400/500/600/700). **JetBrains Mono** for code. Loaded via Google Fonts in `src/index.css`. Six type styles: `h1` (32/40), `h2` (24/30), `h3` (18/24), `t1` (16/22), `t2` (14/20), `t3` (12/16) — exposed as Tailwind utilities `text-h1`…`text-t3` and consumed via the `Text` component. Values are Figma-driven (resolved from `Global.json` typography composites).
- **Radius scale:** `none / 4 / 8 / 12 / 16 / 24 / full`. Default component radius is 8px; cards and modals use 12px.
- **Shadow language:** four-step neutral scale sourced from the Figma file (page "Elevation", node 46814:2892) — `shadow-xs` for chips/tags, `shadow-sm` for tooltips/toasts, `shadow-md` for menus/raised cards, `shadow-lg` for modals and popovers floating over busy content. Single-stop drop shadows; no brand-hue tinting in v1.
- **Motion:** standard ease is a custom cubic-bezier (roughly `cubic-bezier(0.32, 0.72, 0, 1)`). Durations: fast 120ms, base 200ms, slow 320ms. Always respect `prefers-reduced-motion`.
- **Density:** comfortable, not compact. Default control height: 36px (md).

If a design decision doesn't fall out of these, ask before inventing.

## Authoring workflow

New components follow a two-step skill flow in `.claude/skills/` — invoke the skill, don't reinvent it.

1. **Design in Figma** with the `figma-design` skill. Drives `mcp__figma-console__*` to create a page named for the component plus the four-frame contract: `Anatomy` (annotations naming the Radix primitive), `Variants`, `States`, and `Tokens` (sticker sheet of variables consumed). Every fill/stroke/radius/spacing must bind to a Figma variable.
2. **Generate code** with the `figma-to-code` skill. Reads the Figma page via `figma_get_component_for_development_deep`, maps Figma variable names to Tailwind utilities backed by CSS variables, and writes `<Name>.tsx`, `<Name>.stories.tsx`, `index.ts`, and updates `src/index.ts`. Walks the "Definition of done" checklist before declaring the component finished.

Tokens are Figma-led (see "Tokens" above). If a needed token is missing, add it in Figma first via Tokens Studio, push, run `npm run tokens:build`, then proceed with `figma-design` / `figma-to-code`.

`figma-selection-to-code` is retained for emergency use — every shipped component already has tokens behind it, so new work should use the strict pair.

## Skills

Project-level skills live in `.claude/skills/` and trigger automatically based on phrasing.

- **`figma-design`** — strict Figma authoring. Every value binds to a Figma variable; stops if a needed token is missing.
- **`figma-to-code`** — strict code generation from a named Figma component. Maps Figma variable names to CSS-var-backed Tailwind utilities.
- **`figma-selection-to-code`** _(legacy)_ — emit code from a selected Figma node, with a decisions log of any raw values used. Use only if the strict path is genuinely blocked.

## Working style — how to collaborate with me

- **Token + variable changes are still high-stakes.** Edits to `src/tokens/source/` should go through Tokens Studio; if you must hand-edit, flag it and reconcile with Figma after. Never edit `src/tokens/dist/`.
- **Always read `CLAUDE.md` and the most similar existing component before writing a new one.** Match the patterns already in the repo. Consistency beats cleverness.
- **Prefer editing existing files over creating new ones.** Don't add helper files, utilities, or abstractions unless I ask.
- **No speculative features.** Don't add props, variants, or states without a clear ask.
- **Use plan mode for any task that touches more than one file.** I want to see the plan before code is written.
- **When uncertain about an API decision, ask.** Don't pick silently. The most similar shipped component is the fallback reference for "what would users expect?"
- **Don't run `npm install` for new dependencies without asking.** Every new dep is a maintenance cost.
- **Tests are now expected for new components and bug fixes.** v1 stabilized faster than originally planned, so the "no tests yet" stance is retired. Test runner selection (Vitest expected) is pending — see open work below.
- **Comments are sparse and useful.** Don't narrate what the code does; only explain _why_ when it isn't obvious.

## Definition of "done" for a component

A component is done when **all** of these are true:

- [ ] Implementation matches the Figma source spec (variants, sizes, states)
- [ ] Built on the appropriate Radix primitive (where one exists)
- [ ] Uses cva for variants, no inline ternaries on classNames
- [ ] All visual properties come from tokens (no raw colors/spacing/radii/shadows)
- [ ] `forwardRef` correctly typed; `...rest` spread to root
- [ ] Props type exported alongside the component
- [ ] Storybook story exists with: Default, Variants, Sizes, States, Playground
- [ ] Storybook a11y addon shows zero violations
- [ ] Visible focus ring via `focus-visible:`
- [ ] Renders correctly in **all three themes** — toggle light/dark/black via the Storybook toolbar to verify
- [ ] Exported from `src/index.ts`
- [ ] Co-located test file (once Vitest lands)

## Things to never do

- Never edit files in `src/tokens/dist/` directly.
- Never use inline styles (`style={{ ... }}`) for theming. CSS variables and Tailwind utilities only. Inline styles are acceptable only for dynamic values that genuinely can't be expressed as classes (e.g., a computed pixel offset).
- Never hardcode `#hex` colors, raw px values for spacing, or raw shadow strings in component code. Per-component theming uses `data-theme` on the root, not hex literals per branch (see Architecture rules #8).
- Never reach into another component's internals or import from a sibling component's non-`index.ts`.
- Never add a dependency without asking first.
- Never auto-accept changes to `package.json`, `tsconfig.json`, `tailwind.config.*`, `vite.config.ts`, or anything in `src/tokens/`. These are load-bearing — show me the diff and wait.
